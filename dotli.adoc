== Executive Summary

`dot.li` is a universal resolver linking Polkadot Name Service (dotNS) names to on-chain content. A user types `name.dot.li` into any browser and receives the associated decentralized website or application.

The service operates via two resolution methods:

. **Client-Side (Primary):** A lightweight Universal Viewer app runs in the user's browser, using embedded light clients to query dotNS and fetch content from the Bulletin Chain directly. No intermediary touches the data.
. **Server-Side (Fallback):** A traditional gateway server resolves names and proxies content for crawlers, bots, and legacy clients.

This is `eth.limo` for Polkadot, but more decentralized.

== Problem Statement

[cols="1,3"]
|===
| Problem | Impact

| dApp frontends are centralized
| Single point of failure and censorship risk undermines the entire value proposition of decentralized backends.

| On-chain content is inaccessible
| Users cannot access Bulletin Chain or IPFS content without special tools or long, ugly CID strings.

| dotNS names are underutilized
| `.dot` names have no browsable interface, limiting their utility to wallet address resolution.
|===

== Solution Architecture

=== Resolution Methods

==== Client-Side Resolution (Trustless)

This is the primary and preferred method.

[source]
----
User Browser                       Polkadot Network
     |                                        |
     |  1. Request: parityproduct.dot.li/about
     |--------------------------------------->|
     |                                        |
     |  2. Serve Universal Viewer (static HTML/JS)
     |<---------------------------------------|
     |                                        |
     |  3. Viewer JS: Parse URL -> "parityproduct.dot", "/about"
     |                                        |
     |  4. Light Client: Query dotNS for _manifest record
     |--------------------------------------->|
     |                                        |
     |  5. Return: Manifest CID
     |<---------------------------------------|
     |                                        |
     |  6. Light Client: Fetch Manifest from Bulletin Chain
     |--------------------------------------->|
     |                                        |
     |  7. Return: Manifest JSON { "/about": "CID_about" }
     |<---------------------------------------|
     |                                        |
     |  8. Light Client: Fetch CID_about from Bulletin Chain
     |--------------------------------------->|
     |                                        |
     |  9. Return: Content (HTML/JS/CSS)
     |<---------------------------------------|
     |                                        |
     | 10. Render content in sandboxed <iframe>
     |
----

==== Server-Side Resolution (Fallback)

Used for SEO crawlers, social media link previews, and legacy clients.

[source]
----
Client                        dot.li Gateway              Polkadot / Storage
  |                                 |                            |
  | 1. GET parityproduct.dot.li/about                            |
  |-------------------------------->|                            |
  |                                 |                            |
  |                                 | 2. Query dotNS: contenthash
  |                                 |--------------------------->|
  |                                 |                            |
  |                                 | 3. Return: ROOT_CID        |
  |                                 |<---------------------------|
  |                                 |                            |
  |                                 | 4. Fetch: ROOT_CID/about   |
  |                                 |--------------------------->|
  |                                 |                            |
  |                                 | 5. Return: Content         |
  |                                 |<---------------------------|
  |                                 |                            |
  | 6. Proxy content to client      |                            |
  |<--------------------------------|                            |
  |                                 |                            |
----

== Manifest Specification v1.0

The Manifest is a JSON file stored on-chain that defines application structure and routing.

=== Schema

[source,json]
----
{
  "manifest_version": "1.0",
  "app_type": "website",
  "root": "<CID>",
  "paths": {
    "<path>": "<CID>"
  },
  "spa_fallback": true
}
----

=== Field Definitions

[cols="2,1,4"]
|===
| Field | Type | Description

| `manifest_version`
| string
| Specification version. Must be `"1.0"` for this version.

| `app_type`
| enum
| Content type hint for the Universal Viewer. Values: `website`, `widget`, `profile_card`.

| `root`
| string
| CID of the default content (e.g., `index.html`). Served when path is `/` or on SPA fallback.

| `paths`
| object
| Key-value mapping of URL paths to content CIDs. Example: `{ "/about": "bafy...abc" }`.

| `spa_fallback`
| boolean
| If `true`, any path not found in `paths` resolves to `root` CID. Required for SPAs.
|===

=== Example

[source,json]
----
{
  "manifest_version": "1.0",
  "app_type": "website",
  "root": "bafybeigdyrzt5sfwedcva2mlt7gzx5ryfpcifpwafwazljo4l4y",
  "paths": {
    "/about": "bafybeihkoviema7g3gxyt6la7vd5ho32bsc5",
    "/blog": "bafybeigeq5vdhgltnyv5xsked7gh66a7dc",
    "/assets/logo.png": "bafkreiabcdef123456789"
  },
  "spa_fallback": true
}
----

=== Resolution Logic

[source]
----
function resolve(path, manifest):
    if path in manifest.paths:
        return manifest.paths[path]
    else if manifest.spa_fallback == true:
        return manifest.root
    else:
        return 404_NOT_FOUND
----
